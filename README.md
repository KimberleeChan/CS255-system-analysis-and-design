## CS 255 Module Eight Journal Reflection

### Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?

The DriverPass project involved designing a system for a client who wanted to help students prepare for their driving tests. The client was DriverPass, a company that provides online and in-person driving test preparation services. They wanted a system that would allow users to create accounts, purchase packages, schedule driving lessons, take practice tests, and access training materials. The goal was to build a web-based system that would help students prepare for their driving tests by giving them practice exams and a way to schedule driving lessons.

### What did you do particularly well?

I think I did well at identifying and organizing the system requirements in a clear way. I was able to separate the functional and nonfunctional requirements so it was easier to see what the system needed to do versus how it needed to perform. I also did a good job turning those requirements into a system design that included the main parts of the system like user access, scheduling features, and data management.

### If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could revise one part, I would make the diagrams more consistent and a bit clearer overall. Some of the interactions between users and the system, especially around scheduling and checking instructor availability, could be shown in more detail so the flow is easier to follow. That would make the design easier to understand and more complete.

### How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?

I interpreted the user’s needs by focusing on what the client wanted the system to do from the user’s point of view, like making scheduling easy, managing accounts, and providing access to learning materials. I then turned those needs into specific features and requirements for the system. It’s important to consider user needs because the system only works well if it actually solves their problems and is easy to use. Even if the system works technically, it won’t be effective if it isn’t designed with the user in mind.

### How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?

My approach to software design starts with understanding the problem and collecting clear requirements. From there, I break the system into smaller parts so it’s easier to understand what each feature does and how it connects. I then figure out how the parts interact so the system makes sense as a whole. In the future, I would use tools like use case analysis, UML diagrams, flowcharts, and separating functional and nonfunctional requirements to help keep everything organized. I would also focus more on iterative design, where I improve the system through feedback and testing instead of getting everything fully correct on the first attempt.
